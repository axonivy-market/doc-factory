Repository Organisation

The IvyAddOns repository is organised in a standard subversion way with a trunk for the current development a branches directory where hotfixes for older versions are maintained and a tags directory where tags of the releases can be found. E.g:
DirectoryDescription
trunkThe current development branch. Development in this branch will be included in the next Xpert.ivy release (Xpert.ivy trunk). Therefore use the latest available development build of Xpert.ivy to develop in the trunk.
branches/4.2The current development branch for Xpert.ivy 4.2. Use the latest available HotFix Version of Xpert.ivy 4.2 to develop things here.
tags/4.2.xTags of the different IvyAddOns 4.2 Versions. The IvyAddOns Versions are independent of Xpert.ivy HotFix Versions. Only major and minor of IvyAddOns and Xpert.ivy must be equal.
tags/4.2.latestReleaseSpecial tag that is linked (subversion externals) with the Xpert.ivy development repository. The contents of this tag is overtaken to the Xpert.ivy4.2 and its build infrastructure. The version commited here is automatically overtaken to the next Xpert.ivy 4.2 Hotfix.
tags/trunk.latestStableSpecial tag that is linked (subversion externals) with the Xpert.ivy development repository. The contents of this tag is overtaken to the latest Xpert.ivy development. The version commited here is automatically overtaken to the next Xpert.ivy Sprint Release.

Subersion externals can be seen as mirrors or links. The Xpert.ivy repository links to the special tags described above:

This means that any changes commited to these tags are automatically overtaken to Xpert.ivy and its build infrastructure. The idea behind this is that developers can work on the trunk or on branches until they reached a stable version which they want to release. They can release them by copying the trunk to the trunk.latestStable tag or the branch 4.2 to the 4.2.latestRelease tag:
Note that hotfixes or development made on any branch have to be merged or copied to the trunk.
